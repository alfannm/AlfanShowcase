App 1 — JiranLink

Flutter Firebase Geolocation

Problem

Residential neighborhoods suffer from resource redundancy (underutilized household tools) and lack a trusted, hyper-local platform for borrowing and renting items.

Solution

  • Peer-to-Peer Marketplace: A dual-interface system allowing users to switch between Owner and Borrower roles to rent or share equipment.
  • Smart Discovery: Implements geo-location services to automatically filter search results by the user's current State and District.
  • Booking Lifecycle: A complete transaction flow from request to payment and item return, managed via a status-tracking system.

Engineering Decisions

  • State Management: Used the Provider pattern to manage shared state across complex widget trees (Auth, Bookings, Messages).
  • Backend: Built on Google Firebase using Firestore for real-time data, Auth for user management, and Storage for media assets.
  • Architecture: Modular design with a consistent top-down structure (MaterialApp → MultiProvider → AuthGate).

Challenges & Improvements

  • Challenge: Ensuring location accuracy and filtering relevance for hyper-local queries.
  • Next: Enhance scalability with advanced geo-querying techniques and implement pagination for listing feeds.

App 2 — FuelWise

Android Native Java / XML Room DB

Problem

Vehicle owners lack a centralized way to track fuel expenses, leading to inaccurate manual calculations and difficulty in identifying vehicle performance issues.

Solution

  • Digital Logbook: Replaces paper receipts with a centralized digital record for multiple vehicles.
  • Auto-Calculation: Automatically computes key metrics like Fuel Efficiency (L/100km) and Cost-per-Kilometer (RM/km) to eliminate human error.
  • Vehicle Profiles: Allows management of distinct profiles for different vehicles, each with its own history and statistics.

Engineering Decisions

  • Architecture: Implemented MVVM (Model-View-ViewModel) to separate the UI from business logic and ensure data persistence during configuration changes.
  • Data Layer: Used the Room Persistence Library (SQLite abstraction) for robust local storage and offline capability.
  • Performance: Executed all database write operations on background threads using ExecutorService to prevent UI blocking.

Challenges & Improvements

  • Challenge: Enforcing data integrity, such as preventing users from entering mileage lower than previous records.
  • Next: Add data visualization (charts) for consumption trends and cloud backup for data security.

App 3 — RideShare Buddy

Flutter FluidBackgroundPainter

Problem

Students face prohibitive costs from commercial e-hailing services and acute parking congestion at university campuses.

Solution

  • Trusted Network: A "Walled Garden" ecosystem that requires Matric/Staff ID verification to ensure safety and trust.
  • Dual-Stream Feed: Features separate feeds for "Ride Offers" and "Ride Requests" to efficiently match drivers with passengers.
  • Campus-Specific Logic: Custom filtering based on specific university landmarks (e.g., Kolej Kediaman, Library).

Engineering Decisions

  • UI Optimization: Utilized ListView.builder for lazy-loading to maintain high performance with large lists of ride offers.
  • Component Reuse: Created a reusable RideCard widget that adapts its appearance (color/layout) for both offers and requests to reduce code duplication.
  • State Management: Leveraged StatefulWidget to handle dynamic updates for filters and profile forms.

Challenges & Improvements

  • Challenge: Managing state consistency when switching between filter options and dynamic lists.
  • Next: Integrate a real-time in-app chat (replacing the current WhatsApp link) and a digital payment gateway.
